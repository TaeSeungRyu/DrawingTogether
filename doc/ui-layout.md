# UI 화면 레이아웃

전체 화면 흐름은 `splash → home → draw | pairing → draw`. 패키지/네비게이션 구조는 [architecture.md](architecture.md), 어떤 화면이 어느 Phase인지는 [roadmap.md](roadmap.md).

## 1. 설계 원칙

- **세션 액션은 상단, 그리기 도구는 하단** — TopAppBar는 "무엇을 그리는가"(사진/내보내기), 도구바는 "어떻게 그리는가"(붓/도형/색). 영역 충돌 회피.
- **싱글과 멀티는 같은 화면** — 차이는 TopAppBar 우측 peer indicator 하나뿐. 코드 재사용 + 사용자 학습 비용 최소.
- **사진은 캔버스의 상태** — 액션은 한 곳(TopAppBar)에 모이고, 도구바는 매번 그리기 동작만.

## 2. 홈 화면 (Phase 1.6 갱신)

```
┌──────────────────────────────────┐
│                                  │
│        DrawingTogether           │
│        모드를 선택해 주세요         │
│                                  │
│  ┌────────────────────────────┐  │
│  │         싱글모드             │  │
│  │         혼자 그리기          │  │
│  └────────────────────────────┘  │
│                                  │
│  ┌────────────────────────────┐  │
│  │         멀티모드             │  │
│  │   가까운 기기와 함께 그리기   │  │
│  └────────────────────────────┘  │
│                                  │
│  ─────────────                   │
│  최근 작품                        │
│  [▣][▣][▣][▣][▣]  ─────►          │ 가로 스크롤 (LazyRow)
│                                  │
└──────────────────────────────────┘
```

"가까운 기기와 함께 그리기" — 멀티모드 라벨 (Nearby 반영). 작품 없을 땐 "아직 저장된 작품이 없어요" 안내. 썸네일은 92×92dp, `ContentScale.Crop`. 탭 시 `preview/{workId}` 라우트로 이동.

## 3. 드로잉 화면 — 싱글 모드 (Phase 1.5+)

```
┌──────────────────────────────────────┐
│ ←  📷사진  📸촬영  ⤴PNG       ✕제거 │ ← TopAppBar (56dp)
├──────────────────────────────────────┤
│                                      │
│                                      │
│            [Canvas]                  │ ← 사진 있으면 사진 비율 (Fit)
│      사진 배경 + stroke 오버레이       │   없으면 흰 배경
│                                      │
│                                      │
├──────────────────────────────────────┤
│ [● ● ● ● ● ● ● ● ● ● ● ● [+]]     │ 색 팔레트
│ [붓: 펜 ▼] [모양: 자유 ▼] [지우개]   │
│ 굵기 4dp  [────●─────]                │
│                   [되돌리기][전체지우기]│
└──────────────────────────────────────┘
```

### TopAppBar 액션

| 액션 | 동작 | Phase | 가시성 | 색 |
|---|---|---|---|---|
| ← | 홈으로 (백 스택 pop) | 1 | 항상 | 기본 |
| 사진 | Android PhotoPicker — 갤러리 선택, 권한 불필요 | 1.5 | 항상 | primaryContainer |
| 촬영 | `ACTION_IMAGE_CAPTURE` 인텐트(FileProvider URI), CAMERA 권한 미선언 | 1.5 | 항상 | tertiaryContainer |
| 제거 | `canvas.setBackground(null)` | 1.5 | 사진 있을 때만 | errorContainer |
| 저장 | `PngComposer.compose` → `WorkStore.save` → Toast | 1.6 | 항상 | secondaryContainer |

### 캔버스 영역

- **사진 없음**: 흰 배경, 캔버스 가로 폭 가득, 1:1 정사각형 영역.
- **사진 있음**: 사진 종횡비 유지 (`ContentScale.Fit`). letterbox 영역(여백)은 `surfaceVariant` 톤. letterbox 영역은 **입력 비활성** — stroke는 사진 영역 안에서만.

## 4. 드로잉 화면 — 멀티 모드 (Phase 2-3)

```
┌──────────────────────────────────────┐
│ ←  📷사진  📸촬영  ⤴PNG  ✕제거  🟢kim│ ← peer indicator 추가
├──────────────────────────────────────┤
│                                      │
│        [Canvas — 양쪽 동기화]          │
│                                      │
├──────────────────────────────────────┤
│ [도구바 동일]                          │
└──────────────────────────────────────┘
```

싱글 모드와의 차이:
- **우측 끝에 peer indicator** — 연결 상태 + 닉네임
- **사진 선택/촬영이 자동으로 상대에게 전송** — 별도 "보내기" 버튼 없음
- 사진 전송 중 진행률은 Snackbar 또는 indicator 옆 작은 ProgressIndicator

### 액션 영역 레이아웃 (TopAppBar `actions`)

좁은 폰(예: 갤럭시 S21, 360dp)에서 액션이 한 줄에 다 안 들어가는 문제는 **가로 스크롤 + peer indicator 고정** 으로 해결한다. 연결 상태는 항상 보여야 하므로 같은 스크롤 그룹에 두면 안 됨.

```kotlin
actions = {
    Row(modifier = Modifier.weight(1f).horizontalScroll(...)) {
        // 배경합치기 토글, 사진, 촬영, 제거, 저장
    }
    PeerIndicator(...)   // Phase 2 — 항상 우측 고정
}
```

싱글 모드는 `PeerIndicator` 호출이 없어 weight(1f) Row가 액션 영역 전체를 차지. 멀티 모드는 indicator가 우측 슬롯을 차지하고 액션 행이 좁아진 만큼 스크롤로 흡수.

### Peer Indicator 상태

| 상태 | 아이콘 | 의미 |
|---|---|---|
| Connected | 🟢 | 정상 동기화 중 |
| Transferring | 🟡 | 사진 전송 진행 중 (진행률 같이 표시) |
| Slow | 🟠 | PING 응답 지연 |
| Disconnected | 🔴 | 끊김 — 재연결 다이얼로그 |

## 6. 페어링 화면 (Phase 2)

```
┌──────────────────────────────────────┐
│ ←  멀티모드                           │
├──────────────────────────────────────┤
│                                      │
│  내 이름: [ ryu___________ ]           │
│                                      │
│  ┌──────────────────────────────┐    │
│  │  📡 호스트로 광고시작         │    │
│  └──────────────────────────────┘    │
│                                      │
│   또는 발견된 디바이스:                │
│   ┌────────────────────────┐         │
│   │ 🟦 kim                  │ ← 탭   │
│   └────────────────────────┘         │
│   ┌────────────────────────┐         │
│   │ 🟦 lee                  │        │
│   └────────────────────────┘         │
│                                      │
│   ⓘ Nearby가 BT/Wi-Fi 자동 선택       │
│                                      │
└──────────────────────────────────────┘
```

### 흐름

1. 화면 진입 → 양쪽 자동으로 `startDiscovery` (백그라운드).
2. 한쪽이 "호스트로 광고시작" 탭 → 그쪽은 `startAdvertising` 모드.
3. 다른쪽 화면에 광고하는 디바이스 카드 등장 → 탭.
4. 양쪽 화면에 **인증 다이얼로그** 등장: "kim과 연결할까요? 토큰: 4839". 양쪽 수락 시 연결 성립.
5. 자동으로 드로잉 화면 이동: `navigate("draw") { popUpTo("pairing") }`.

### 권한 흐름

화면 진입 직후 한 번에 묶어 요청 — 자세한 권한 매트릭스는 [nearby-connections.md](nearby-connections.md) §3.
거부 시 "권한이 없어 멀티모드를 사용할 수 없습니다" 안내 + 홈 복귀.

## 7. 화면 비율과 공간

폰 700dp 세로 기준:

| 영역 | 높이 |
|---|---|
| 상태바 | ~28dp |
| TopAppBar | 56dp |
| 캔버스 | ~386dp |
| 도구바 | ~230dp |

작은 폰에서는 도구바가 답답해질 수 있음. 보류 옵션:
- 도구바 접기 (▼ 버튼으로 미니 도구바 모드)
- 가로 모드: 도구바를 우측 패널로 이동

실측 후 필요해지면 추가.

## 8. 디자인 결정 — 왜 TopAppBar인가

| 후보 | 장점 | 단점 |
|---|---|---|
| **A. TopAppBar (선택)** | 표준 Material 패턴, 도구바와 영역 분리 | 56dp 캔버스 손실 |
| B. 도구바 4번째 행 | 캔버스 공간 절약 | 5행 빽빽함, 그리기/세션 액션 혼재 |
| C. FAB (떠있는 메뉴) | 캔버스 chrome 최소 | 캔버스 가림, 다중 액션 묶기 어려움 |
| D. 햄버거 메뉴 | 가장 적은 chrome | 사진 접근에 2탭 — 핵심 액션엔 부적합 |

A 선택 이유: 사진/내보내기는 "보이는 곳에" 있어야 할 핵심 액션. 도구바와의 영역 분리가 인지적으로 깔끔.
