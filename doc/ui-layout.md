# UI 화면 레이아웃

전체 화면 흐름은 `splash → home → draw | pairing → draw`. 패키지/네비게이션 구조는 [architecture.md](architecture.md), 어떤 화면이 어느 Phase인지는 [roadmap.md](roadmap.md).

## 1. 설계 원칙

- **세션 액션은 상단, 그리기 도구는 하단** — TopAppBar는 "무엇을 그리는가"(사진/내보내기), 도구바는 "어떻게 그리는가"(붓/도형/색). 영역 충돌 회피.
- **싱글과 멀티는 같은 화면** — 차이는 연결 시 캔버스 우측 상단에 뜨는 내 닉네임 칩뿐. 코드 재사용 + 사용자 학습 비용 최소.
- **사진은 캔버스의 상태** — 액션은 한 곳(TopAppBar)에 모이고, 도구바는 매번 그리기 동작만.

## 3. 드로잉 화면 — 싱글 모드 (Phase 1.5+)

```
┌──────────────────────────────────────┐
│ ←  [📷사진][📸촬영][🎨배경색][▭비율][🖼트레이싱][🗑제거][⤓저장] │ ← TopAppBar (아이콘+라벨, 가로 스크롤)
├──────────────────────────────────────┤
│                                      │
│                                      │
│            [Canvas]                  │ ← 사진 있으면 사진 비율, 없으면 CanvasAspect
│   배경색 + 사진 + stroke 오버레이      │   (두 손가락 핀치/이동으로 줌)
│                              [⤢150%] │ ← 줌 리셋 칩(확대 시 좌하단, 탭→1:1)
│                                      │
├──────────────────────────────────────┤
│ [●●●●●●● [+][💧스포이드] | ◐◐◐ 최근색] │ 색 팔레트 (가로 스크롤)
│ [✏️붓][◆도형][⭐스티커][🩹지우개][▦보조] │  ← 도구 아이콘 5등분(한 줄 고정)
│ 굵기 4dp  [────●─────]    [보정 끔]    │  ← 보정 칩(끔→약→강). 세밀붓이면 "굵기 고정"
│                   [되돌리기][전체지우기]│  ← 넘치면 가로 스크롤(우측 정렬)
└──────────────────────────────────────┘
```

> 도구 행은 라벨 달린 아이콘 5개를 `weight(1f)` 로 균등 배치 — 가로 스크롤 없이 항상 전부 보임.
> 각 아이콘은 선택 시 강조되고, 탭하면 세부 시트/드롭다운(붓 종류·도형·스티커·**보조**)이 열린다.
> "보조" 드롭다운은 **안내선**(중앙 십자선·격자)과 **대칭**(끔/좌우/상하/4분할) 두 섹션을 묶는다.
> 붓은 미선택 시 펜 자유 모드로, 이미 선택돼 있으면 붓 종류 시트를 연다. 붓 종류엔 **세밀붓**(굵기 고정, 세밀 묘사용) 포함 — 선택 시 굵기 슬라이더가 "굵기 고정" 안내로 대체된다.
> 도형 드롭다운 하단엔 **채우기 토글**(외곽선 ↔ 색 채움, `ToolSettings.fill`).
> 색 팔레트 줄: 프리셋 7색 + 사용자 정의(+) + **스포이드 버튼** + 구분선 뒤 **최근 사용 색**(프리셋 중복 제외).
> 굵기 줄 우측 **보정 칩**은 손떨림 보정 끔→약→강을 순환 토글.
> TopAppBar **배경색** 버튼은 현재 배경색 스와치로 표시(항상), **비율** 버튼(`AspectRatioSheet`)·**트레이싱** 버튼은 사진 없을 때만/사진 있을 때만 각각 노출.
> **줌**: 두 손가락 핀치로 확대(최대 5배)+드래그 이동, 한 손가락은 그리기(확대 상태에서도 손끝 아래 정확히). 확대 중 좌하단 **줌 리셋 칩**(⤢N%) 탭 시 1:1. 로컬 표시 전용 — 상대·저장엔 미반영.

### TopAppBar 액션

액션은 **아이콘 + 작은 라벨** 버튼(`TopActionButton`). Material 확장 아이콘 의존성 없이
직접 그린 글리프(`PhotoGlyph`/`CameraGlyph`/`TrashGlyph`/`SaveGlyph`, 하단 도구 줄과 같은 방식).
좁은 폭 대비 가로 스크롤(+페이드) 유지.

| 액션 | 글리프 | 동작 | Phase | 가시성 | 색 |
|---|---|---|---|---|---|
| ← | ArrowBack | 홈으로 (백 스택 pop) | 1 | 항상 | 기본 |
| 사진 | 액자+해+산 | Android PhotoPicker — 갤러리 선택, 권한 불필요 | 1.5 | 항상 | primaryContainer |
| 촬영 | 카메라 | `ACTION_IMAGE_CAPTURE` 인텐트(FileProvider URI), CAMERA 권한 미선언 | 1.5 | 항상 | tertiaryContainer |
| 배경색 | 색 채운 액자 | `ColorPickerSheet` → `canvas.setBackgroundColor` | 5.5 | 항상 | **현재 배경색 스와치**(밝기 따라 글자 대비 자동) |
| 비율 | 사각형 글리프 | `AspectRatioSheet` → `canvas.setCanvasAspect`(+연결 시 `CanvasAspectFrame` 송신) | 이후 | **사진 없을 때만** | secondaryContainer |
| 트레이싱 | 사진+펜 | 사진 표시 알파 순환(원본/연하게/아주 연하게/**외곽선**, `TraceOpacity`) | 5.5 | 사진 있을 때만 | tertiaryContainer(활성 시 tertiary) |
| 제거 | 휴지통 | `canvas.setBackground(null)` | 1.5 | 사진 있을 때만 | errorContainer |
| 저장 | 다운로드 화살표 | `PngComposer.compose` → `WorkStore.save` → Toast | 1.6 | 항상 | secondaryContainer |
| 기록/종료 | 빨간 점 / 정지 | 타임랩스 기록 토글 — 종료 시 `TimelapseStore.save`. 기록 중 캔버스 좌상단 ● REC m:ss(15분 소프트 가드) | 5.5 | 항상 | 기록=primaryContainer / 종료=errorContainer |

> **"사진 배경 포함" 토글**(구 "배경 합치기")은 상단 바에서 빼고 **저장 다이얼로그**로 이동(저장
> 시에만 의미). **사진 배경이 있을 때만 활성**, 없으면 흐리게 disabled. 함께 모드(Duo) 연결 중이면
> `Frame.MergeBackground` 로 동기화. 모임 모드는 자기만(브로드캐스트 X).

### 캔버스 영역

- **사진 없음**: 흰 배경, 캔버스 가로 폭 가득, 1:1 정사각형 영역.
- **사진 있음**: 사진 종횡비 유지 (`ContentScale.Fit`). letterbox 영역(여백)은 `surfaceVariant` 톤. letterbox 영역은 **입력 비활성** — stroke는 사진 영역 안에서만.

## 4. 드로잉 화면 — 함께 모드 (Phase 2-3)

```
┌──────────────────────────────────────┐
│ ←   [📷사진][📸촬영][🗑제거][⤓저장]    │ ← TopAppBar (indicator 없음)
├──────────────────────────────────────┤
│                              [회색 내이름]│ ← 연결 시 캔버스 우측 상단 닉네임 칩
│        [Canvas — 양쪽 동기화]          │
│                                      │
├──────────────────────────────────────┤
│ [도구바 동일]                          │
└──────────────────────────────────────┘
```

싱글 모드와의 차이:
- **연결 시 캔버스 우측 상단에 내 닉네임 칩** (반투명 surface + 회색 글자) — 아래 §"연결 상태 표시" 참고
- **사진 선택/촬영이 자동으로 상대에게 전송** — 별도 "보내기" 버튼 없음
- 사진 전송 중 진행률은 로딩 오버레이(`TransferLoadingOverlay`)

### 액션 영역 레이아웃 (TopAppBar `actions`)

좁은 폰(예: 갤럭시 S21, 360dp)에서 액션이 한 줄에 다 안 들어가는 문제는 **가로 스크롤** 로 해결한다.

```kotlin
actions = {
    Row(modifier = Modifier.weight(1f).horizontalScroll(...)) {
        // 배경합치기 토글, 사진, 촬영, 제거, 저장
    }
}
```

### 연결 상태 표시 — 캔버스 내 닉네임

상단 바의 peer indicator(상대 닉네임 + 🟢)는 제거됨. 대신 **연결되면 내 캔버스 우측 상단에
내 닉네임을 반투명 칩**(surface 65% + 둥근 모서리 + 회색 글자)으로 표시한다(`MyCanvasContent`
의 `selfNick` 오버레이). 칩 배경 덕에 사진·색 있는 그림 위에서도 읽힌다. `pointerInput` 이
없어 그리기 터치에 영향 없음. 싱글 모드(미연결)는 표시하지 않는다.

### 5.3 레이아웃 규칙 요약

| 항목 | 세로 | 가로 |
|---|---|---|
| 미니 뷰 방향 | 캔버스 아래, 가로 분할 | 캔버스 우측, 세로 스택 |
| 미니 뷰 영역 차지율 | 약 25% (수직) | 약 25% (수평) |
| N=0 일 때 | 미니 영역 숨김, 안내문 표시 | 우측 칼럼 숨김 |
| N=1 일 때 | 미니 1개가 가로 전체 | 미니 1개가 세로 전체 |
| N=2 일 때 | 미니 2개 1/2씩 가로 분할 | 미니 2개 1/2씩 세로 분할 |
| N=3 일 때 | 미니 3개 1/3씩 가로 분할 | 미니 3개 1/3씩 세로 분할 |
| 도구바 위치 | 최하단 (전체 폭) | 최하단 (전체 폭, 가로 영역 활용) |

### 5.4 Compose 구현 가이드

```kotlin
val cfg = LocalConfiguration.current
val isLandscape = cfg.orientation == Configuration.ORIENTATION_LANDSCAPE
val peers: List<PeerInfo> = ... // SessionManager 가 노출

Column(Modifier.fillMaxSize()) {
    TopAppBar(...)
    if (isLandscape) {
        Row(Modifier.weight(1f)) {
            DrawingCanvas(..., Modifier.weight(3f))   // 좌측 75%
            if (peers.isNotEmpty()) {
                Column(Modifier.weight(1f)) {           // 우측 25% 세로 스택
                    peers.forEach { peer ->
                        MiniCanvas(peer, Modifier.weight(1f))
                    }
                }
            }
        }
    } else {
        DrawingCanvas(..., Modifier.weight(3f))         // 상단 75%
        if (peers.isNotEmpty()) {
            Row(Modifier.weight(1f)) {                  // 하단 25% 가로 분할
                peers.forEach { peer ->
                    MiniCanvas(peer, Modifier.weight(1f))
                }
            }
        } else {
            EmptyPeerPlaceholder(...)
        }
    }
    Toolbar(...)
}
```

핵심: **`Modifier.weight()` 가 자동으로 동등 분할** — 1/2/3 명 모두 같은 코드. `peers.forEach` 만으로 케이스가 흡수됨.

### 멀티(1:1) 모드와의 차이

| 항목 | 함께 (1:1, Duo) | 모임 (1:N, Party) |
|---|---|---|
| Strategy | `P2P_POINT_TO_POINT` | `P2P_STAR` (호스트 + 조인자 최대 3) |
| 캔버스 | 1개 (공유) | 1개 (내 것) + 미니 뷰 N 개 |
| 자기 사진 배경 | 양방향 자동 broadcast → 양쪽 메인 캔버스 동기화 | 자기 메인 + **다른 사람들이 보는 내 미니 뷰**에 표시. 상대 메인엔 영향 없음 |
| 사진 제거 | 양쪽 동기화 | 자기 메인 + 다른 사람의 내 미니 뷰에서 사라짐 |
| 배경 합치기 토글 | 양방향 broadcast | 자기만 (저장 옵션) |
| 미니 뷰의 사진 배경 | (해당 없음, 1개 공유 캔버스) | 그 peer 의 사진 + stroke 둘 다 표시 |
| 캔버스 비율 | 사진 있으면 사진 비율, 없으면 `CanvasAspect`(동기화됨) | 자기 메인·미니 뷰 모두 그 peer 의 사진 비율 또는 `CanvasAspect`(동기화) |
| Clear/Undo/지우개 | 모두 영향 (함께 그리기) | 내 캔버스만 |
| "동기화" | 즉시 컨펌 → SnapshotReq (사진 포함) | 다이얼로그 → 누구 캔버스 가져올지 선택 → 컨펌 (사진 안내) → 타겟 SnapshotReq (호스트 relay). 받은 뒤 자기 캔버스를 다시 broadcast 해 다른 사람의 내 미니 뷰도 갱신 |
| 끊김 1명 | 세션 종료 | 조인자 끊김은 그 미니 뷰만 `EmptyMiniSlot` 으로. 호스트는 "방 열기" 로 재모집 가능. (호스트 끊김=전체 종료는 미구현 — 보류) |

> 사진 정책 변경 이력: 초기 설계는 "자기 사진은 자기만" (미니 뷰 미표시) 이었으나, 사용자 피드백으로 "자기 사진을 다른 사람들이 보는 내 미니 뷰에도 표시" 로 반전됨. 상대 *메인* 캔버스엔 여전히 안 들어감 — 어디까지나 그 사람의 미니 뷰에만.

### 미니 뷰 구성

- `DrawingCanvas` 와 동일 렌더 함수 재사용 (`StrokeRenderer.drawStroke`) — 화면 = 저장 일관성.
- `pointerInput` 미부착 → 탭/드래그 모두 무시.
- 그 peer 의 사진 배경 + stroke 둘 다 렌더 (배경은 `MiniCanvas` 가 직접 그림).
- 슬롯 안에 1:1 정사각형 letterbox (`aspectRatio(1f)`) — 가로/세로 어느 슬롯 방향이든 일관.
- 닉네임 라벨 상단 표시 (peer 식별).
- 빈 슬롯은 `EmptyMiniSlot` — "비어있음" placeholder. 슬롯은 항상 3개 고정 (참가자 수와 무관).

### "동기화" 선택 다이얼로그

2단계로 진행:

**1단계 — 누구 캔버스 가져올지 선택**
```
┌──────────────────────────────┐
│  어느 사람의 그림을 가져올까요? │
│                              │
│  ◯ kim                       │
│  ◯ lee                       │
│  ◯ park                      │
│                              │
│      [취소]      [다음]       │
└──────────────────────────────┘
```

**2단계 — 컨펌**
```
┌──────────────────────────────────┐
│  동기화                          │
│                                  │
│  상대방 데이터를 전부 가져와       │
│  적용 합니다.                    │
│  내가 그린 그림은 전부 제거됩니다.  │
│                                  │
│  ⓘ 상대방 사진이 있는 경우        │
│     사진 정보도 가져옵니다.       │
│                                  │
│      [취소]        [적용]        │
└──────────────────────────────────┘
```

- 적용 시 선택한 피어에게 타겟 `SnapshotReq(targetPeerId, requesterPeerId)` (호스트 relay 거침)
- 응답(`Snapshot` + strokes FILE) 도착 → 내 메인 캔버스 stroke 덮어쓰기
- 응답에 사진 포함되면 → `PhotoMeta` + FILE 페이로드 따라옴 → 내 캔버스 배경에 적용
- **strokes + 사진 적용이 끝난 뒤** 내 캔버스를 `targetPeerId=""` 로 다시 broadcast → 호스트 relay → 다른 참가자가 보는 *내 미니 뷰*도 가져온 내용으로 갱신. (사진 적용 전에 broadcast 하면 빈 배경이 전송되므로 순서 중요.)
- 함께(Duo) 모드의 sync-with-photo FILE 매칭 메커니즘 재사용

### 호스트/조인 비대칭 점

- 호스트 화면: 동작 같음 (조인자 미니 뷰 N개)
- 조인자 화면: 동작 같음 (호스트 + 다른 조인자 미니 뷰)
- 차이는 transport 내부 — 조인자→조인자 메시지는 호스트가 relay (P2P_STAR 한계)

  MP4 → `MediaStore.Video`(Movies/DrawingTogether) + 공유. 진행률 오버레이.
- 삭제: 재생 화면 상단 또는 갤러리 길게 누르기.

```

│                                      │

│  │  📡 호스트로 광고시작         │    │
│  └──────────────────────────────┘    │
│                                      │
│   또는 발견된 디바이스:                │
│   ┌────────────────────────┐         │
│   │ 🟦 kim                  │ ← 탭   │
│   └────────────────────────┘         │
│   ┌────────────────────────┐         │
│   │ 🟦 lee                  │        │
│   └────────────────────────┘         │
│                                      │
│   ⓘ Nearby가 BT/Wi-Fi 자동 선택       │
│                                      │
└──────────────────────────────────────┘
```

### 흐름

1. 화면 진입 → 양쪽 자동으로 `startDiscovery` (백그라운드).
2. 한쪽이 "호스트로 광고시작" 탭 → 그쪽은 `startAdvertising` 모드.
3. 다른쪽 화면에 광고하는 디바이스 카드 등장 → 탭.
4. 양쪽 화면에 **인증 다이얼로그** 등장: "kim과 연결할까요? 토큰: 4839". 양쪽 수락 시 연결 성립.
5. 자동으로 드로잉 화면 이동: `navigate("draw") { popUpTo("pairing") }`.

## 8. 화면 비율과 공간

폰 700dp 세로 기준:

| 영역 | 높이 |
|---|---|
| 상태바 | ~28dp |
| TopAppBar | 56dp |
| 캔버스 | ~386dp |
| 도구바 | ~230dp |

작은 폰에서는 도구바가 답답해질 수 있음. 보류 옵션:
- 도구바 접기 (▼ 버튼으로 미니 도구바 모드)
- 가로 모드: 도구바를 우측 패널로 이동

실측 후 필요해지면 추가.

## 9. 디자인 결정 — 왜 TopAppBar인가

| 후보 | 장점 | 단점 |
|---|---|---|
| **A. TopAppBar (선택)** | 표준 Material 패턴, 도구바와 영역 분리 | 56dp 캔버스 손실 |
| B. 도구바 4번째 행 | 캔버스 공간 절약 | 5행 빽빽함, 그리기/세션 액션 혼재 |
| C. FAB (떠있는 메뉴) | 캔버스 chrome 최소 | 캔버스 가림, 다중 액션 묶기 어려움 |
| D. 햄버거 메뉴 | 가장 적은 chrome | 사진 접근에 2탭 — 핵심 액션엔 부적합 |

A 선택 이유: 사진/내보내기는 "보이는 곳에" 있어야 할 핵심 액션. 도구바와의 영역 분리가 인지적으로 깔끔.
