## 1. 나눠 그리기 (Split Drawing) — 신규 모드

홈 화면 신규 모드. 2~4명이 **하나의 합성 캔버스를 레이아웃으로 나눠** 각자 자기 구역을 그린다.

### 요구사항 (사용자 스펙)
- **대상**: 2~4명.
- **레이아웃**(인원별 선택지):
  - 2명: 상하, 좌우
  - 3명: 상하 3단, 좌우 3분할, 상1·하좌우2, 상좌우2·하1
  - 4명: 상하 4단, 좌우 4분할, 2×2
- **미리보기 버튼**: 누르면 n명이 그린 그림이 정해진 레이아웃대로 **1개 모달에 합성**되어 보임.
- **저장 버튼**: 전체 합성 그림을 하나로 저장.

### 기존 인프라와의 관계
모임 모드(Party, `P2P_STAR` + `peerCanvases` + 실시간 mesh 동기화 + 늦은참여 자동채움)가 이미
"N명이 각자 캔버스 + 서로의 캔버스를 모두 보유"를 제공한다. 나눠 그리기는 **Party 위에
"레이아웃 슬롯 배정 + 합성 미리보기/저장"을 얹는** 형태 — 전송·동기화 계층은 대부분 재사용.

**재사용**: Party 전송/동기화, 늦은참여 채움, `MiniCanvas`, `PngComposer`, `CanvasAspect`.

### 핵심 설계 결정 (착수 시 확정)
- **각자 무엇을 그리나** — (후보) 각자 자기 전체 캔버스에 그리고, 미리보기/저장 시 각 캔버스를
  배정 슬롯 dstRect에 합성. 슬롯 비율을 `CanvasAspect`로 강제하면 합성이 깔끔(예 좌우 2분할 →
  각자 세로로 길쭉한 비율).
- **레이아웃·슬롯 배정 프로토콜** — 호스트가 레이아웃+슬롯을 정해 broadcast(기존 `PartyStart`
  확장 또는 신규 Frame). 인원별 레이아웃 목록은 UI 상수.
- **라이브 뷰 범위** — 스펙대로 "평소엔 자기 슬롯만, 미리보기 버튼 → 합성 모달". 모두의 캔버스는
  이미 `peerCanvases`에 실시간 존재하므로 모달은 그걸 슬롯에 배치만.
- **합성 저장** — `PngComposer`에 "각 캔버스를 슬롯 rect에 그리는" 변형 추가. 미리보기 모달도
  같은 합성 재사용. 저장 해상도 기준 결정 필요.

### 주 리스크
- 슬롯 비율 ↔ 각 참가자 캔버스 비율 정합(비율 강제로 해결).
- 저장 해상도/각 슬롯 픽셀 배분.

## 2. 작업 데이터 저장/로드 (편집 가능 프로젝트)

PNG(그림)로만이 아니라 **데이터로 저장**해, 나중에 불러와 지우개/삭제/undo 등 **편집**이 되게 한다.

### 기존 인프라와의 관계
데이터 모델·직렬화가 **이미 존재**한다:
- `CanvasSnapshot`(strokes + stickers + texts + aspect + backgroundColor) CBOR 직렬화
  (동기화 FILE 페이로드·타임랩스에서 사용).
- `CanvasState.applySnapshot(...)`로 복원.
- **#8 수정(요소 `seq` 병합정렬)** 덕분에 로드 후 undo 순서가 시간순으로 정확.
- `WorkStore` / 타임랩스 저장(`filesDir`, CBOR 로그 + 배경 png) 패턴 참고 가능.

### 고려사항 (사용자 제기) — 방향
- **이미지(사진 배경)** — `CanvasSnapshot`엔 사진 비트맵이 없음(사진은 캔버스 속성, 도메인
  이벤트 아님). 타임랩스가 `bg-<n>.png`를 별도 저장하듯, 편집본 = `<id>.canvas`(CBOR) +
  `<id>-bg.png`(선택) + `<id>.png`(썸네일) + `<id>.meta` 묶음.
- **작업 기록 삭제** — `WorkStore.delete` 패턴 그대로(그 id의 파일 전부 제거 + 인덱스 갱신).
- **작업 기록 공유** — 두 갈래:
  - (a) 보기용: 기존 PNG 공유(`ACTION_SEND` + FileProvider).
  - (b) 편집용: CBOR 번들(+사진)을 하나의 "프로젝트 파일"로 export/import. (a) 먼저, (b)는 후속.

### 핵심 설계 결정 (착수 시 확정)
- **"작품(PNG 갤러리)" vs 별도 "프로젝트" 개념** — (후보) 저장 다이얼로그에 "편집 가능하게 저장"
  옵션 → 같은 store에 `.canvas` 동반 저장, 미리보기 화면에 "이어서 편집" 버튼.
- **버전 호환** — 모델 변경돼도 옛 저장본이 열리게 `default` 필드 + `ignoreUnknownKeys` 패턴 유지.

### 주 리스크
- 포맷 버전 관리(장기 저장물의 하위 호환).
- 사진 배경 용량(원본 대신 다운샘플 저장).

## 상대적 난이도 / 순서 제안
- **1번(나눠 그리기)**: Party 재사용도 높음 → 상대적으로 빠름. 핵심은 레이아웃/슬롯/합성.
- **2번(프로젝트 저장/로드)**: 데이터·직렬화가 이미 있어 저장·로드·이미지 배선이 핵심.

둘 다 착수 전 이 문서에서 결정 포인트를 확정한 뒤 플랜 모드로 상세화한다.
