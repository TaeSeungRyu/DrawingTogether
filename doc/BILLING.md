# 수익화 메모 (보류 — 나중에 검토)

> 상태: **미구현·미결정.** 아이디어 보관용. 기능 안정화가 우선.

## 발단

"모임 모드 / 함께 모드에 일일 이용 횟수 제한(각 최대 3회)을 걸고, 인앱결제로
일일 20회 / 무제한을 파는" 모델 검토.

## 결론 요약

**횟수 제한 → 결제 해제** 모델은 이 앱 카테고리에서 전환율이 낮고 이탈을 부를
가능성이 큼. 의미 있는 매출은 어렵다고 판단. 단, "친구와 함께"라는 소셜 훅은
강점이라 바이럴로 사용자를 모으고 수익화는 다른 방식이 유리.

### 횟수 제한 모델의 구조적 약점

- **협업은 양쪽이 다 써야 성립** — 한 명이 횟수를 소진하면 상대도 같이 못 함.
  횟수 제한이 네트워크 효과를 거꾸로 깎음. ("친구랑 그리려는데 너 횟수 다 썼네"
  → 둘 다 이탈)
- **무료 경쟁 치열** — 그리기/화이트보드 앱은 대체재가 많아 "왜 횟수를 사야 하지?"
  라고 느끼는 순간 이탈.
- **일 3회는 캐주얼엔 충분, 헤비엔 적음** — 가벼운 사용자는 결제 동기 없고,
  무제한 필요한 헤비 유저는 소수라 매출 베이스가 얇음.

### 더 가능성 있는 방향 (대안)

- **기능 기반 부가 판매(비소모성 IAP)** — 프리미엄 브러시팩, 색 팔레트 슬롯 확장,
  큰/고해상도 캔버스, 워터마크 제거, 저장 무제한(현재 `WorkStore.MAX_WORKS=100`
  제한과 연결). "쓰는 걸 막기"보다 "더 좋게" 파는 쪽이 거부감 적음.
- **모임 모드 인원 제한** — 무료 2~3인, 결제 시 4인+ (횟수보다 자연스러운 업셀).
- **광고(전면/리워드) + 광고 제거 1회 구매** — 캐주얼 앱의 흔한 베이스라인.

## 기술 메모 (구현하게 되면)

- **Google Play Billing Library** 사용. 소모성(consumable) vs 비소모성
  (non-consumable) vs 구독(subscription) 구분:
  - 일일 횟수 충전 = 소모성, 광고 제거/기능 해제 = 비소모성, 무제한 = 구독.
- 횟수 카운터를 둔다면 **로컬 prefs 만으론 위변조 쉬움** — 재설치/시간 조작으로
  리셋 가능. 서버 검증 없이는 신뢰 못 함. (서버리스 유지가 이 앱의 강점인데
  결제 검증 때문에 서버가 필요해지는 비용도 고려.)
- 현재 앱은 **완전 서버리스**(Nearby P2P + 로컬 저장). 결제·횟수 검증을 제대로
  하려면 백엔드가 생기는데, 그 운영비 대비 매출이 나올지부터 따져야 함.
- 가장 운영 부담 적은 시작점: **비소모성 "프리미엄 언락" 1건** (Play 가 영수증
  보관, 서버 불필요). 기능 팩부터.

## 판단

지금은 보류. 사용자 수가 의미 있게 모이고(바이럴 확인) 나서, 그때 "기능 팩 +
광고 제거" 같은 거부감 적은 모델부터 실험. 횟수 제한 판매는 비추천.
